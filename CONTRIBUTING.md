# Contributing to Solarplan

Thank you for your interest in improving the solarplan project! This document provides guidelines for contributing.

## Ways to Contribute

### Report Issues or Suggest Improvements
- **Found an error or ambiguity?** Open a [bug report](../../issues/new?template=bug_report.md)
- **Have a suggestion?** Open an [improvement issue](../../issues/new?template=improvement.md)
- **Questions about the system?** Start a [Discussion](../../discussions)

### Improve the Documentation
- Clarify unclear sections
- Add diagrams or images
- Document real-world installations or modifications
- Share commissioning tips and troubleshooting
- Correct errors or outdated information

### Share Real-World Experience
If you've built this system or a variation:
- Document your installation process
- Share lessons learned
- Report performance data and efficiency results
- Suggest alternative components with equivalent specs

## How to Contribute

### For Issues and Suggestions
1. Check existing [issues](../../issues) to avoid duplicates
2. Use the appropriate issue template:
   - [Bug Report](../../issues/new?template=bug_report.md) for errors or unclear information
   - [Improvement](../../issues/new?template=improvement.md) for suggestions
3. Provide as much detail as possible
4. For component suggestions, include specifications and sourcing

### For Documentation Updates
1. Fork the repository
2. Create a feature branch: `git checkout -b improve/your-description`
3. Make your changes
4. Test that the Markdown renders correctly
5. Commit with a clear message: `git commit -m "Improve: description of change"`
6. Push to your fork and open a pull request

## Pull Request Process

1. **Describe the change**: What does this improve? Why is it needed?
2. **Stay focused**: Keep PRs scoped to one improvement
3. **Maintain style**: Keep the formatting and tone consistent with existing content
4. **Reference issues**: Link to any related issues: "Fixes #123"
5. **Be patient**: Reviews may take a few days

## Style Guide

- **Headings**: Use Markdown heading levels (# ## ###) hierarchically
- **Code blocks**: Use triple backticks with language tags (```mermaid, etc.)
- **Lists**: Use clear, parallel structure
- **Links**: Use relative paths for internal docs, full URLs for external resources
- **Marine Terms**: Use consistent terminology (bank, bus, run, loom, etc.)
- **Technical Specs**: Include units and part numbers for component references

## Standards & Requirements

When suggesting components or modifications:
- Prioritize **marine-grade certified** parts
- Verify **UL or equivalent** safety certifications
- Use **tinned copper** for all bilge-area cabling (UL 1426)
- Ensure **voltage and amperage ratings** match the system
- Document **sourcing and cost** for Bill of Materials updates

## Questions?

- Check the [README](README.md) and [ABOUT](ABOUT.md) for project overview
- Review the [CHANGELOG](CHANGELOG.md) for recent updates
- Open a [Discussion](../../discussions) if you have questions

## Recognition

Contributors are recognized in the project. If you make significant contributions, we'll add your name to the contributors list.

## Code of Conduct

This project follows a simple principle: **be respectful, collaborative, and focused on improving the project**. Everyone is welcome.
